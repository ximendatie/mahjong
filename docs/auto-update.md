# Auto-Update Strategy

mahjong 1.x should make updates easier without weakening the local-first trust
model. The current decision is to defer Sparkle integration until signed and
notarized release credentials are available for routine builds.

## Current 1.x Decision

- 1.0 and early 1.x releases use manual updates from GitHub Releases.
- Release artifacts remain zip and dmg files generated by the existing scripts.
- Sparkle is the preferred auto-update candidate once Developer ID signing,
  notarization, and release-note publishing are routine.

## Why Sparkle Is Deferred

- Unsigned or ad-hoc signed update feeds create more user trust friction than
  they remove.
- Automatic update metadata needs a stable signing and hosting process.
- The app is still early enough that manual release notes are useful for
  explaining provider and privacy changes.

## Manual Update Path

1. Download the latest dmg or zip from GitHub Releases.
2. Quit mahjong from the menu bar or Dock.
3. Replace the previous `mahjong.app` with the new build.
4. Launch mahjong again and verify Provider Diagnostics.

## Sparkle Readiness Checklist

- [ ] Developer ID signing is configured for release builds.
- [ ] Notarization is part of the release workflow.
- [ ] The update feed can be generated and hosted reliably.
- [ ] Release notes clearly describe provider, permission, and privacy changes.
- [ ] Update prompts are optional and low-noise.
- [ ] Failed updates leave the existing app usable.
